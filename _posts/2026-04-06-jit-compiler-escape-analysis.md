### 1. 객체지향의 딜레마와 메모리 성능

프로젝트를 진행하며 도메인 주도 설계(DDD)와 불변 객체 패턴을 적용하다 보니, 상태가 변경될 때마다 새로운 객체를 생성해 반환하는 방식을 자주 사용하게 되었습니다. 코드는 비교적 안정성을 확보하고 가독성이 높아졌지만, 대규모 트래픽을 가정하면 자연스럽게 성능에 대한 고민이 뒤따랐습니다.

'수천 번 반복되는 루프 안에서 매번 new 키워드로 단명 객체를 생성해도 괜찮을까? 잦은 힙 메모리 할당과 GC에 의한 Stop-The-World 현상이 시스템의 병목을 유발하지 않을까?' 하는 고민이었습니다.

과거에는 이러한 오버헤드를 피하고자 오브젝트 풀 패턴을 사용하거나 객체지향 설계를 포기하고 원시 타입 변수들을 나열하는 방어적 프로그래밍을 하기도 했습니다. 하지만 대규모 트래픽을 다루는 서비스 환경을 고려할 때, 유지보수성을 포기하고 성능만 좇는 것은 장기적으로 리스크가 큽니다.

그래서 '객체를 많이 생성하면 무조건 느려진다'는 추측에 기반한 우려로 설계 원칙을 타협하는 대신, 코드가 실제로 구동되는 JVM 런타임 내부를 직접 분석하기로 했습니다.

<br>

### 2. JVM의 판단: 탈출 분석과 스칼라 치환

자바의 모든 객체는 힙에 생성된다고 알려져 있지만, 현대 JVM의 C2 JIT 컴파일러 환경에서는 항상 성립하는 명제가 아닙니다. JIT 컴파일러는 코드의 표면적인 형태를 넘어 런타임 실행 흐름을 추적하는 탈출 분석을 수행합니다.

탈출 분석은 특정 메서드 내에서 생성된 객체가 해당 스코프 밖으로 벗어나는지를 판단합니다.

- **Global-Escape (전역 탈출):** 객체가 정적 필드나 메서드 반환값으로 사용되어 외부에서 접근 가능한 상태.
- **Arg-Escape (인자 탈출):** 객체가 다른 메서드의 인자로 전달되는 상태.
- **No-Escape (탈출 불가):** 객체가 생성된 메서드 내부에서만 소비되는 상태.

컴파일러가 특정 객체를 No-Escape로 판별할 경우 JVM은 객체를 스택에 올리는 것을 넘어 객체가 가진 필드들을 원시 타입(Scalar) 단위로 쪼개어 CPU 레지스터나 지역 변수로 분산시킵니다. 이를 스칼라 치환이라고 부릅니다.

> <img width="1650" height="354" alt="image" src="https://github.com/user-attachments/assets/fe7cc8b5-0559-442f-b9bf-39666fe98775" />
> 
> *스칼라 치환이 적용되면 객체의 헤더가 제거되고 필드 데이터만 레지스터에 할당됩니다.*

결과적으로 힙에 객체가 생성되지 않으므로, GC가 수거할 대상 자체도 발생하지 않게 됩니다.

<br>

### 3. JMH 마이크로 벤치마크 실험

스칼라 치환이 성능과 메모리에 미치는 영향을 확인하기 위해 JMH(Java Microbenchmark Harness)를 활용하여 실험 환경을 구축했습니다.

```java
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.NANOSECONDS)
@State(Scope.Thread)
public class EscapeAnalysisBenchmark {

    static class Point {
        int x, y;
        Point(int x, int y) { this.x = x; this.y = y; }
    }

    private int a = 10;
    private int b = 20;

    @Benchmark
    public void testScalarReplacement(Blackhole blackhole) {
        // Point 객체는 메서드 내부에서만 생성 및 소비됨 (No-Escape)
        Point p = new Point(a, b);
        int result = p.x + p.y;
        
        // 원시 타입인 result만 Blackhole에 소비시켜 참조 탈출 방지
        blackhole.consume(result); 
    }
}
```

JMH의 `-prof gc` 옵션을 활용하여 탈출 분석 최적화를 명시적으로 껐을 때(Case A)와 켰을 때(Case B)의 지표를 추출했습니다.

| **실험 케이스** | **JVM Options** | **평균 응답 시간** | **작업당 힙 할당량 (B/op)** |
| --- | --- | --- | --- |
| **Case A (최적화 OFF)** | `-XX:-DoEscapeAnalysis` | `3.541 ns/op` | `24.000 B/op` |
| **Case B (최적화 ON)** | `-XX:+DoEscapeAnalysis` | `0.979 ns/op` | `≈ 10⁻⁶ B/op` |

최적화를 비활성화한 Case A에서는 Point 객체가 생성될 때마다 메타데이터와 필드를 포함해 약 24바이트의 단명 객체가 Eden 영역에 쌓였습니다. 반면 탈출 분석이 활성화된 Case B에서는 할당률이 사실상 0바이트로 수렴했습니다. 객체 생성이 생략되어 메모리 접근 비용이 절감되었고, 결과적으로 응답 시간도 향상되었습니다.

### 4. 기계어 수준의 확인: 어셈블리 코드 분석

벤치마크 수치 외에 CPU가 실제로 수행하는 명령어를 확인하고자 `hsdis` 플러그인을 활용하여 JIT 컴파일러가 생성한 x86 어셈블리 코드를 분석해 보았습니다.


**최적화 이전: 힙 할당 방식 (Escape Analysis OFF)**

최적화가 꺼진 상태에서는 객체를 생성하기 위해 TLAB(Thread Local Allocation Buffer) 여유 공간을 확인하고 객체 헤더를 설정하는 과정을 거칩니다.

```assembly
; TLAB 공간 확인
0x00007f...: mov    r10, QWORD PTR[r15+0x60]   
0x00007f...: lea    r11, [r10+0x18]             ; 24바이트(헤더+필드) 할당 계산
0x00007f...: cmp    r11, QWORD PTR [r15+0x70]   ; 공간 부족 시 느린 할당 호출

; 객체 헤더(Mark Word, Klass Pointer) 및 필드 세팅
0x00007f...: mov    QWORD PTR [r10], 0x1        ; Mark Word 초기화
0x00007f...: mov    DWORD PTR [r10+0x8], 0x...  ; Klass Pointer 설정
0x00007f...: mov    DWORD PTR[r10+0xc], r8d    ; x 값 할당
0x00007f...: mov    DWORD PTR[r10+0x10], r9d   ; y 값 할당
```

`new Point()`라는 코드 이면에서 메모리 공간을 계산하고 클래스 메타데이터를 설정하는 등 부가적인 관리 비용이 발생하고 있음을 알 수 있습니다.


**최적화 이후: 스칼라 치환 적용 (Escape Analysis ON)**

반면 탈출 분석이 적용된 코드는 다음과 같이 변경됩니다.

```assembly
; 객체 할당 및 헤더 설정 코드 소거
0x00007f...: mov    eax, DWORD PTR [r12+0x10]   ; 변수 a를 eax 레지스터로 로드
0x00007f...: add    eax, DWORD PTR[r12+0x14]   ; 변수 b를 더함 (x + y)
```

힙 메모리 할당(`lea`, `cmp`)과 관련된 명령어들이 생략되었습니다. 컴파일러는 `Point`라는 객체의 구조가 런타임에 불필요하다고 판단하고 내부 값만 CPU 레지스터(`eax`)로 가져와 덧셈 연산을 수행합니다.

<br>

### 5. 스칼라 치환이 실패하는 경우

하지만 JIT 컴파일러의 최적화가 모든 상황에 적용되는 것은 아닙니다. 코드를 작성하는 방식에 따라 탈출 분석이 실패하여 객체가 힙에 할당되는 경우도 존재합니다.

1. **객체 크기가 클 때:** 치환해야 할 필드가 많아 레지스터나 스택 프레임에 담기 어려운 경우 최적화를 포기합니다. (보통 64바이트 초과 등 JVM 옵션에 종속)
2. **제어 흐름이 복잡할 때:** 객체가 생성되는 위치와 사용되는 위치 사이에 복잡한 루프나 조건문이 얽혀 있어, 컴파일러가 생명주기를 정적으로 분석하기 어려운 경우입니다.

코드가 지나치게 복잡하거나 객체 크기가 비대하다면, 컴파일러의 최적화 이점을 누리기 어렵습니다.

<br>

### 6. 결론: 런타임 환경을 고려한 코드 작성

이번 분석을 통해 애플리케이션 레벨에서 메모리를 아끼고자 억지로 객체를 재사용하거나 코드를 복잡하게 만드는 것이 오히려 컴파일러의 최적화를 방해할 수 있음을 확인했습니다.

성능을 확보하기 위해 무작정 객체 생성을 피하기보다는, 메서드를 작고 응집도 있게 유지하여 JIT 컴파일러가 분석하기 좋은 코드를 작성하는 것이 실용적인 접근일 수 있습니다. 대규모 트래픽 환경의 시스템 아키텍처를 설계할 때, 단순히 프레임워크의 기능을 활용하는 것을 넘어 코드가 실행되는 런타임 환경의 특성을 이해하는 것이 중요하다고 생각합니다.
